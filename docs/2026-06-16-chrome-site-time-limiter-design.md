# Chrome Site Time Limiter Design

## 提炼总结

第一版做成 Chrome 扩展，用来限制用户在知乎等指定网站上的浏览时间。扩展默认监控 `zhihu.com`，用户可以在设置页添加或删除其他域名。计时只在 Chrome 当前活跃窗口的当前活跃标签页命中监控域名时进行，切走、浏览器失焦、休眠或访问非监控网站时不计时。

扩展提供三种监控模式：

1. 温和模式：单次连续浏览到时后关闭当前标签页。
2. 拦截模式：每日总时长用完后，所有受监控网站页面跳转到扩展内的额度用完页面。
3. 防绕过模式：在拦截模式基础上，调松限制、删除站点或切换到更温和模式需要延迟生效。

第一版包含设置页、拦截页、后台计时服务、通知提醒和本地存储。时间数据按本地日期每日重置。先不做跨浏览器、云同步、手机端和操作系统级防卸载。

## 目标

- 帮用户减少在知乎及类似网站上的无意识长时间浏览。
- 支持用户自己维护受监控网站列表，而不是只写死知乎。
- 支持从温和到较强的三档约束，适配不同自控需求。
- 保持第一版足够简单，能作为可安装、可测试的 Chrome 扩展运行。

## 非目标

- 不监控手机 App。
- 不支持 Chrome 之外的浏览器。
- 不做系统级拦截、防卸载或家长控制。
- 不上传用户浏览数据。
- 不做账号体系或云同步。

## 用户体验

用户安装扩展后，设置页显示：

- 监控网站列表，默认包含 `zhihu.com`。
- 添加域名输入框，允许添加如 `weibo.com`、`youtube.com`。
- 单次连续浏览时长，默认建议 15 分钟。
- 每日总时长，默认建议 45 分钟。
- 监控模式选择：温和、拦截、防绕过。
- 今日已用时长展示。

当用户访问受监控网站时，后台开始累计本次连续浏览时长和今日总时长。接近限制时，扩展发送 Chrome 通知。达到限制后，根据当前模式执行动作。

温和模式下，单次时长到达后关闭当前受监控标签页。每日总时长仍继续记录并提醒，但不做全站拦截。

拦截模式下，每日总时长到达后，任何受监控网站标签页都会跳转到扩展内的 blocked 页面。blocked 页面显示今日额度已用完、已用时长和明天重置提示。

防绕过模式下，达到每日总时长后的表现与拦截模式相同。此外，用户如果尝试删除受监控站点、调大时长或切换到更温和模式，这些改动进入待生效状态，默认 10 分钟后生效。调严限制可以立即生效。

## 架构

扩展使用 Chrome Manifest V3。

主要组成：

- `manifest.json`：声明权限、后台 service worker、设置页、通知权限和拦截页资源。
- `background.js`：负责标签页状态监听、活跃窗口监听、计时、通知、模式动作和按日重置。
- `options.html` / `options.js`：设置页，管理网站列表、时长、模式和今日统计展示。
- `blocked.html` / `blocked.js`：额度用完页面。
- `storage.js`：封装 Chrome storage 读写和默认配置。
- `domain.js`：负责 URL 与受监控域名匹配。

## 数据模型

配置数据保存在 `chrome.storage.local`：

```json
{
  "settings": {
    "sites": ["zhihu.com"],
    "sessionLimitMinutes": 15,
    "dailyLimitMinutes": 45,
    "mode": "block",
    "warningMinutesBefore": 2,
    "strictDelayMinutes": 10
  },
  "usage": {
    "date": "2026-06-16",
    "dailySeconds": 0,
    "activeSite": null,
    "sessionStartedAt": null,
    "sessionSeconds": 0
  },
  "pendingSettings": null
}
```

模式枚举：

- `gentle`
- `block`
- `strict`

## 域名匹配

用户输入规范化为裸域名，例如：

- `https://www.zhihu.com/question/123` 规范化为 `zhihu.com`
- `www.zhihu.com` 规范化为 `zhihu.com`
- `zhihu.com` 保持为 `zhihu.com`

匹配规则：

- `zhihu.com` 命中 `zhihu.com`
- `www.zhihu.com` 命中 `zhihu.com`
- `api.zhihu.com` 命中 `zhihu.com`
- `notzhihu.com` 不命中 `zhihu.com`

## 计时规则

后台 service worker 每隔固定间隔检查当前状态，建议 1 秒或 5 秒一跳。第一版可用 5 秒降低复杂度和资源消耗。

计时条件全部满足时才累计：

- Chrome 当前窗口处于焦点状态。
- 当前窗口的活跃标签页存在。
- 活跃标签页 URL 命中受监控网站。
- 当前日期与 usage 日期一致；如果不一致，先重置今日数据。

当活跃受监控站点改变时，当前 session 重新开始。切到非受监控网站或 Chrome 失焦时，session 暂停。

## 动作规则

提醒：

- 单次时长距离限制小于等于 `warningMinutesBefore` 时，通知一次。
- 每日总时长距离限制小于等于 `warningMinutesBefore` 时，通知一次。
- 每个日期、每个限制类型只通知一次，避免重复打扰。

温和模式：

- 单次时长达到 `sessionLimitMinutes` 后，关闭当前受监控标签页。

拦截模式：

- 每日总时长达到 `dailyLimitMinutes` 后，将当前以及后续所有受监控标签页跳转到 `blocked.html`。

防绕过模式：

- 每日总时长达到 `dailyLimitMinutes` 后，与拦截模式相同。
- 调严设置立即生效。
- 调松设置保存为 `pendingSettings`，在 `strictDelayMinutes` 后生效。

## 错误处理

- URL 无法解析时，不计时、不拦截。
- Chrome storage 读取失败时，使用默认设置并在设置页显示轻量错误提示。
- 通知权限不可用时，不阻塞计时和拦截。
- 标签页关闭失败时，降级为跳转到 blocked 页面。

## 测试策略

手动测试：

- 安装未打包扩展。
- 访问 `zhihu.com`，确认今日计时增加。
- 切到非监控网站，确认计时停止。
- Chrome 失焦，确认计时停止。
- 添加新域名，确认新域名被计时。
- 温和模式下缩短单次限制，确认到时关闭标签。
- 拦截模式下缩短每日限制，确认受监控页面跳转到 blocked 页面。
- 防绕过模式下尝试调松设置，确认进入待生效状态。

单元测试优先覆盖：

- 域名规范化。
- 域名匹配。
- 日期重置。
- 模式动作决策。
- 防绕过模式下的设置变更分类。

## 沟通全文要点

用户提出：经常刷知乎停不下来，希望监控是否在刷知乎，计时，快到时提示，到时强制关闭，并且每天有总时长，超过后不再允许看。

经过澄清，范围确定为电脑端，并且主要使用 Chrome。初始建议是做 Chrome 扩展，而不是 Windows 后台程序或手机端方案。用户选择了“拦截模式”：超过每日总时长后，所有知乎页面跳到“今日额度已用完”的拦截页。

随后用户扩展需求：监控模式需要能在设置里选择温和、拦截、防绕过三种之一；监控网站也不能只限知乎，需要允许用户自行添加其他网站。最终确认第一版先做 Chrome 插件。
